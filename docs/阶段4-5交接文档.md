# QuizPane 共享材料与题组支持 · 阶段 4/5 交接文档

## 0. 前置说明：当前进度

阶段 1（Schema + BankValidator）、阶段 2（DeclarativeProvider 组卷）、阶段 3（MainWindow 材料 UI + DraftStore）已完成，代码在 `dev` 分支（提交 `a499aac`），尚未合并进 `master`。合并是纯 fast-forward，不会冲突。

已落地的关键文件，阶段 4/5 会直接依赖它们：

- `schemas/declarative-provider.schema.json`：单一 Schema（`schemaVersion: 1`），顶层可选 `materials` 数组，题目可选 `materialId`。**没有 v1/v2 版本分裂**——原方案里的"新建 v2 Schema、v1 继续独立运行"的设计已被放弃，改成在唯一的 Schema 里让 `materials` 可选。阶段 4 的所有改动要延续这个约定，不要重新引入版本号分支。
- `core/include/quizpane/bank_validator.hpp` / `core/src/bank_validator.cpp`：`validateBankDetailed()` 已经做材料校验（唯一 ID、分类一致、孤立材料检测、正文不能完整复制进 `stem` 等），`BankValidationError` 已带 `materialId` 字段。
- `sdk/src/declarative_provider.cpp`：`DeclarativeProvider::buildUnits()` 按 `materialId` 分组，`attempt.create` 按组整体组卷，`attempt.questions`/`attempt.solutions` 返回去重后的 `materials` 数组。
- `apps/desktop-qt/src/ui/material_card.{hpp,cpp}` + `main_window.*`：答题页/解析页材料卡片，`materialsById_` 缓存。
- `core/include/quizpane/draft_store.hpp` / `draft_store.cpp`：`DraftSnapshot::materials`，草稿恢复时材料立即可见。

阶段 4、5 要做的事，是让**题库制作器（Question Maker）真正生成**这种带材料的题库，而不是像现在这样只能靠手写 fixture（`schemas/examples/declarative-materials/`）。

参考原始方案全文见 `docs/共享材料与题组支持交接方案.md`（未提交 git，仅本机留存）。下面是结合当前代码现状重新梳理的阶段 4/5 范围。

## 1. 阶段 4：生成引擎 + 检查点升级 + 复核页

### 1.1 目标

现在 `apps/bank-studio/engine/` 下的生成链路（`Chunker` → `GenerationWorkflow` → `CheckpointStore`）完全不认识材料概念：模型提示词只会要求输出 `{"questions":[...]}`，`Chunker` 只按 token 预算切文本，可能把一篇文章和它的子题切到两个不同的请求里。阶段 4 要让这条链路能够：

1. 识别文档里"一篇材料 + 多道子题"的边界，不把它拆开发给模型；
2. 让模型输出 `{"materials":[...], "questions":[...]}`；
3. 校验失败时把材料、关联子题、原文块和错误一起发回模型修复，而不是发单题；
4. 检查点保存材料和已完成题目的关联，支持断点续跑。

### 1.2 涉及文件

| 文件 | 现状 | 要做的改动 |
|---|---|---|
| `apps/bank-studio/engine/include/quizpane/studio/chunker.hpp`<br>`.../src/chunker.cpp` | `Chunker::split()` 只按 token 预算切字符串，返回 `QVector<TextChunk>` | 引入中间结构（方案里叫 `SourceBlock`），先识别"根据以下资料回答 1～5 题" / "材料一 / 材料二" / 连续编号题 + ABCD 选项 / Markdown 标题分隔等强信号，把材料正文和它的子题标记为不可拆分单元，再按 token 预算把普通独立题继续切块 |
| `apps/bank-studio/engine/include/quizpane/studio/generation_workflow.hpp`<br>`.../src/generation_workflow.cpp` | `kSystemPrompt` 只要求 `{"questions":[...]}`；`parseQuestions()` 只解析 `questions` 字段；`handleModelResult()` 校验失败时只把错误消息发回模型，不带材料上下文；`completeChunk()` 只处理 `valid`/`review` 两个题目数组 | 提示词升级为要求 `{"materials":[...], "questions":[...]}`；新增 `parseMaterials()`；修复请求要把材料原文、关联子题、结构化错误一起带上（不能只发单题，否则模型无法修复跨题引用）；`questionsReady` 信号建议按方案 6 节改成传递一个 DTO（例如 `GeneratedBankCandidate { materials, questions, needsReviewQuestions }`），避免继续堆叠平行数组 |
| `apps/bank-studio/engine/include/quizpane/studio/checkpoint_store.hpp`<br>`.../src/checkpoint_store.cpp` | `GenerationCheckpoint` 只有 `completedChunks`（int 列表）+ `questions`/`needsReviewQuestions`，不认识材料 | 检查点结构加 `materials` 字段、`materialId` 的稳定重命名映射、每个材料关联的已完成题目索引；已完成的判定单位从"字符 chunk"改成"源文档块"（对应 `SourceBlock`）。**旧检查点不做迁移**：加载时发现结构对不上，明确报错提示"任务结构已升级，请重新开始"，由用户确认后删除旧检查点，不要猜测旧题目之间的材料关系 |
| `apps/bank-studio/src/studio_window.hpp/.cpp` | `populateReview()` 把 `needsReview` 数组渲染成一个 `QTableWidget` 平铺列表 | 复核页从平铺列表升级成按材料折叠的树形视图（材料标题 + 子题采纳/丢弃）；删除组内全部子题时自动删除材料，保留任意子题时保留材料；材料或引用有结构错误时不允许进入最终打包 |

### 1.3 测试怎么写（不要碰业务代码里的"假模型响应"）

上次已经明确过：**真实模型调用是不确定的，不能进 CTest 断言**。阶段 4 的测试要严格保持"确定性输入 → 确定性输出"的形态，具体到三块：

1. **Chunker**：仿照现有 `tests/chunker_test.cpp` 的写法，喂一段手写的、含"根据以下资料，回答第 1～3 题"强信号的固定文本，断言材料正文和它的题目被分进同一个不可拆分单元，不会跨块。这一步不涉及网络，直接测 `Chunker::split()` 的返回值。
2. **GenerationWorkflow 的修复循环**：这是唯一需要"假模型响应"的地方，且必须做成**测试专属的 mock**，不能改 `ModelClient`/`GenerationWorkflow` 本身的生产代码去适配测试。具体做法：
   - 看 `ModelClient` 现有结构（`apps/bank-studio/engine/include/quizpane/studio/model_client.hpp`）：它通过 `QNetworkAccessManager` 发请求，`finished` 信号带 `GenerationResult`。测试侧可以用 `tests/model_client_test.cpp` 里已经用过的手法——不发真实网络请求，直接构造 `GenerationResult{true, rawJsonText, ...}` 灌给 `handleModelResult()`（如果它是 public 或者能通过信号触发），验证"收到一份 `materialId` 指向不存在材料的响应 → 触发修复请求 → 收到修正后的响应 → 校验通过"这条状态机路径。
   - 如果 `GenerationWorkflow` 当前的方法访问权限不允许这样直接注入，**不要为了测试给生产代码开后门**（比如加一个只有测试用的 public setter）。优先考虑：要么把"校验 + 判断是否需要修复"这一段拆成一个不依赖网络的纯函数单独测（类似把 `parseQuestions`/`bankForQuestions` 这类已经是纯函数的部分抽出来测），要么在测试里用本地 `QTcpServer`/假 HTTP 服务器返回固定 JSON（这是"运行时替换外部依赖"，不是"改业务代码结构去迁就测试"，边界是清楚的）。
3. **CheckpointStore**：跟现有 `tests/checkpoint_store_test.cpp` 的路数一样，手写一个 `GenerationCheckpoint`，测"源文件改了拒绝复用"、"旧结构检查点明确拒绝、不静默迁移"这两个场景。

**红线**：测试 mock 只能存在于 `tests/*.cpp` 里，`GenerationWorkflow`/`ModelClient` 的生产代码不应该出现任何"如果是测试模式就走另一条路径"的分支。如果发现某个改动必须往生产代码里加 `#ifdef TEST` 或者类似的东西才能测，说明设计上有问题，应该先重新拆分职责，而不是妥协。

## 2. 阶段 5：端到端打包与文档

### 2.1 目标

前四个阶段都通过后，做一次完整的端到端验证，并把文档同步更新，作为"正式支持材料题"的收尾。这一步基本不写新业务代码，主要是验证 + 补文档。

### 2.2 具体任务

1. **fixture 复用**：阶段 1 已经有 `schemas/examples/declarative-materials/`（2 份材料 + 6 道子题 + 2 道独立题），端到端测试直接复用，不需要重新造。
2. **真实文档生成压测**（人工验证，不写 CTest 断言）：
   - 方案提到 `build/test-data/c3-real-reading-comprehension.txt`，需要确认这个文件当前是否存在，不存在的话找一份类似的真实篇章阅读材料替代；
   - 用题库制作器实际跑一遍生成流程，人工检查：材料边界识别是否合理、`materialId` 引用是否正确、复核页展示是否符合预期；
   - 不断言固定题数（模型输出非确定），只用于人工质量评估。
3. **打包安装链路**：
   - 生成 `.quizpane-provider` 后过一遍 `ProviderInstaller::inspect()`；
   - 从 ZIP 重新读取，`DeclarativeProvider::load()` 加载成功；
   - 在主程序里走完整流程：进入题组 → 作答 → 交卷 → 查看解析 → 关闭重开验证草稿恢复。
   - 这条链路里 `attempt.create`/`attempt.questions`/`attempt.solutions`/`DraftStore` 都已经在阶段 2/3 有单元测试覆盖，这里主要是"端到端串起来跑一次"，确认没有集成层面的疏漏。
4. **文档更新**：
   - `docs/AI题库生成与声明式Provider方案.md`：补充材料/题组相关的数据流描述（阶段 1 已经把 Schema 文件名从 `declarative-provider-v1.schema.json` 改成了 `declarative-provider.schema.json`，文档已同步这一处，阶段 5 需要把材料相关的生成流程写全）；
   - `docs/架构与Provider开发导读.md`：补充 `materialId` 分组、`buildUnits()` 组卷逻辑在架构图里的位置；
   - `docs/题库生成器UI与集成方案.md`：如果涉及制作器 UI 流程描述，同步复核页的树形视图变化。

### 2.3 完成定义（照抄原方案第 14 节，仍然适用）

- Schema 能无重复地表达共享材料和子题（已完成）；
- 本地校验器能发现所有断裂引用（已完成）；
- 切块不会静默拆散明显题组（阶段 4 待完成）；
- 检查点能恢复材料和关联题目（阶段 4 待完成）；
- Provider 按题组组卷并返回材料（已完成）；
- 主程序独立显示材料并支持草稿恢复（已完成）；
- 打包后的 Provider 能安装、答题和查看解析（阶段 5 待验证）；
