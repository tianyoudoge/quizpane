# QuizPane 共享材料与题组支持交接方案

## 1. 结论与已确认决策

当前声明式题库只能表达彼此独立的题目，不能正确表达“同一篇文章、统计资料或图表贯穿多道子题”的言语理解、资料分析和篇章阅读题组。本项目需要新增原生的共享材料模型，并同步改造 Schema、校验器、生成工作流、检查点、声明式 Provider、组卷逻辑和答题 UI。

本方案采用以下已确认决策：

1. 新建 `declarative-provider-v2`，用顶层 `materials` 和题目 `materialId` 表达共享材料。
2. v1 题库继续按现有逻辑运行；包含共享材料的新题库必须声明 `schemaVersion: 2`。
3. 不做 v2 到 v1 的降级转换。
4. 不把共享材料复制到每一道题的 `stem`。
5. 不在 Provider 或 UI 层临时拼接“材料 + 题干”来伪装支持。
6. 不允许材料引用丢失后静默生成一道人类无法作答的题；必须校验失败或进入定向修复。

“把文章重复塞进每道题题干”的兼容兜底方案已明确放弃。它会造成题库膨胀、切块结果不稳定、复核困难，并掩盖旧运行时不理解共享材料的问题。

## 2. 当前缺口

当前链路的限制不是单点 Bug，而是端到端契约缺失：

- `schemas/declarative-provider-v1.schema.json` 的题目只有 `stem`，没有共享材料或题组引用；
- `core/src/bank_validator.cpp` 不可能校验材料 ID、孤立材料和跨题引用；
- `GenerationWorkflow` 只接收模型返回的 `questions` 数组；
- `Chunker` 只按段落和 token 预算分块，可能把文章和子题切开；
- `CheckpointStore` 只保存题目，不保存共享材料；
- `DeclarativeProvider::hostQuestions()` 只输出题目 `contentHtml`；
- `attempt.create` 按单题抽取，可能拆散一个完整题组；
- `MainWindow` 没有材料区域，也没有材料 ID 到内容的缓存；
- `DraftStore` 只保存当前题目数组，恢复时没有共享材料上下文。

因此，当前模型即使偶然识别出题组，也只能选择重复文章或丢失文章，结果不可作为正式支持。

## 3. v2 数据模型

新增 `schemas/declarative-provider-v2.schema.json`，保留 v1 文件不变。v2 示例：

```json
{
  "schemaVersion": 2,
  "title": "行测题库",
  "catalogs": [
    {
      "id": "verbal",
      "title": "言语理解",
      "practice": { "mode": "sequential", "questionCount": 10 }
    }
  ],
  "materials": [
    {
      "id": "material-001",
      "catalogId": "verbal",
      "title": "阅读材料一",
      "body": "这里是贯穿第 1—5 题的文章正文……",
      "images": [],
      "source": {
        "document": "2025 年行测真题.txt",
        "page": 3
      }
    }
  ],
  "questions": [
    {
      "id": "q-001",
      "catalogId": "verbal",
      "materialId": "material-001",
      "type": "single_choice",
      "stem": "根据材料，下列理解正确的是？",
      "options": [
        { "id": "a", "text": "选项 A" },
        { "id": "b", "text": "选项 B" }
      ],
      "answer": { "optionIds": ["a"] },
      "solution": "解析……"
    }
  ]
}
```

### 3.1 `materials` 字段

`materials` 是可选数组；纯独立题的 v2 题库可以为空。每项建议定义为：

| 字段 | 必填 | 约束 |
|---|---:|---|
| `id` | 是 | 全题库唯一，沿用现有 ID 格式 |
| `catalogId` | 是 | 必须引用现有分类 |
| `title` | 否 | 最长 200 字；UI 可折叠标题 |
| `body` | 条件必填 | 最长建议 100,000 字；与 `images` 至少一个非空 |
| `images` | 条件必填 | 包内资源数组，最多 20 个；与 `body` 至少一个非空 |
| `source` | 否 | 沿用并扩展现有来源信息 |

材料可以是纯文本、纯图片或文本与图片组合。资料分析的表格截图后续可进入 `images`；本阶段 TXT/Markdown 只生成 `body`，不提前实现 PDF/OCR。

### 3.2 `questions[].materialId`

`materialId` 是可选字段：

- 不存在：普通独立题；
- 存在：必须引用同一题库中的材料；
- 题目的 `catalogId` 必须与材料的 `catalogId` 一致；
- 题干只保存子问题，不重复保存材料正文；
- 同一材料关联的题目按 `questions` 数组中的相对顺序展示。

材料本身就是题组身份，不再额外增加含义重叠的 `groupId`。如果未来出现“一份材料包含多个逻辑题组”的需求，再在后续 Schema 版本中扩展，当前不要预先设计双重引用。

### 3.3 版本和 Manifest

共享材料包必须同时声明：

```json
{
  "runtime": {
    "format": "quizpane.bank+json",
    "schemaVersion": 2,
    "entry": "content/bank.json"
  }
}
```

`manifest.runtime.schemaVersion` 必须与 `bank.schemaVersion` 一致。新版安装器和运行时继续接受 v1；旧版应用遇到 v2 必须明确拒绝安装，不能忽略 `materials` 后继续运行。

## 4. 确定性校验规则

在共享 `BankValidator` 中按 Schema 版本分派规则，禁止制作器和 CLI 各写一份：

1. 材料 ID 唯一且格式合法；
2. `catalogId` 必须存在；
3. `body` 和 `images` 至少一个有效；
4. 图片路径仍必须满足包内安全相对路径规则；
5. `materialId` 必须引用真实材料；
6. 题目和材料必须属于同一个分类；
7. 至少一道题引用每份材料，禁止孤立材料；
8. 同一材料关联题目的顺序必须稳定；
9. 不允许将材料正文完整复制进关联题目的 `stem`；测试中至少覆盖明显的完全重复，避免生成器回退到已拒绝方案；
10. v2 Manifest 与 Bank 的 Schema 版本必须一致；
11. v1 校验行为保持不变。

校验错误继续返回结构化错误，但需要增加 `materialId`，使定向修复能明确指出是哪份材料或哪道子题发生引用错误。

## 5. 文档识别与切块

### 5.1 新的中间结构

不要让 `Chunker` 只返回无语义字符串。新增类似以下中间结构：

```cpp
struct SourceBlock {
    enum class Kind { Standalone, MaterialGroup, Unknown };
    QString sourcePath;
    QString heading;
    QString body;
    QStringList questionTexts;
    int startOffset = 0;
    int endOffset = 0;
};
```

先识别文档块，再按 token 预算发送模型。`MaterialGroup` 是不可拆分语义单元。

### 5.2 TXT/Markdown 题组边界

第一阶段至少识别这些强信号：

- “根据以下资料，回答 1～5 题”；
- “材料一 / 材料二 / 阅读下面文字”；
- 连续编号问题和 A/B/C/D 选项；
- Markdown 标题、水平分隔线和明显的“原文/问题/答案”标签；
- C³ 测试文件中的文档分隔符。

不能可靠判断时标记 `Unknown` 并进入模型结构识别与人工复核，不能武断拆分。

### 5.3 超长材料

不能按字符数从中间截断后让不同 chunk 各自生成一部分子题。建议采用两阶段请求：

1. 材料登记：模型读取材料分片，生成稳定的材料摘要/合并结果，不生成题目；
2. 子题结构化：每次携带完整材料或可验证的材料引用，输出关联 `materialId` 的题目。

若完整材料超过模型上下文，应明确失败并提示更换长上下文模型或缩小资料范围。本阶段不实现“只给摘要也宣称能正确答题”的隐式降级。

## 6. 模型输出和修复循环

模型响应从：

```json
{ "questions": [] }
```

升级为：

```json
{ "materials": [], "questions": [] }
```

系统提示必须明确：

- 共享正文只出现一次；
- 每道子题通过 `materialId` 引用；
- 不得把正文复制到 `stem`；
- 普通独立题不设置 `materialId`；
- 不确定题组边界、答案或材料归属时设置 `review.needsReview`。

修复请求必须同时携带材料、关联子题、原文块和结构化校验错误。不能只把单题发回模型，否则模型无法修复跨题引用。

建议把 `questionsReady(questions, needsReview)` 改成传递完整候选题库 DTO，避免后续继续增加平行数组：

```cpp
struct GeneratedBankCandidate {
    QJsonArray materials;
    QJsonArray questions;
    QJsonArray needsReviewQuestions;
};
```

## 7. 检查点升级

检查点版本升级到 v2，至少保存：

- 已确认的 `materials`；
- 已通过校验的普通题和题组子题；
- `materialId` 的稳定重命名映射；
- 已完成的源文档块，而不只是字符 chunk；
- 每个材料关联的已完成题目索引；
- 累计真实 Token；
- 源文件路径和 SHA-256。

旧的 v1 生成检查点不迁移成 v2。加载时明确提示“工作流结构已升级，请重新开始该任务”，由用户确认后删除旧检查点。不要猜测旧题目之间的材料关系。

## 8. 声明式 Provider 与组卷

### 8.1 加载

`DeclarativeProvider::load()` 按版本解析，并建立：

```text
materialId -> material
materialId -> ordered questions
catalogId  -> standalone questions + material groups
```

### 8.2 RPC 输出

`attempt.questions` 不要在每道题里重复材料 HTML。建议返回：

```json
{
  "materials": [
    { "id": "material-001", "title": "阅读材料一", "contentHtml": "..." }
  ],
  "questions": [
    { "id": "q-001", "materialId": "material-001", "contentHtml": "..." }
  ]
}
```

原生 Provider 不使用材料时返回空数组或省略 `materials`，保持现有题目响应可用。

### 8.3 组卷规则

关联同一材料的题目默认是不可拆分的原子题组：

- 顺序模式保持材料和子题的原始顺序；
- 随机模式随机题组，而不是随机打散组内题目；
- 独立题视为只含一道题的单元；
- `questionCount` 是目标题量。若完整加入一个题组会超过目标，优先保证题组完整，最终题量允许超过目标；
- 返回的 `attempt.create.questionCount` 必须是实际题量，而不是请求目标值。

不要为了精确凑够 N 题而截断题组。

## 9. 主程序 UI 与草稿恢复

答题页增加独立材料卡片，位于子题题干上方：

- 支持长文本滚动；
- 支持折叠/展开，但首次进入题组默认展开；
- 同组切换下一题时复用材料控件，不重复解析 HTML；
- 切换到独立题或另一材料时更新/隐藏材料卡片；
- 题号、选项、答案反馈和解析仍属于当前子题；
- 解析页也必须能查看对应材料。

`MainWindow` 建立 `materialId -> material` 缓存。不要在显示题目时把材料 HTML 拼到 `contentHtml`。

`DraftSnapshot` 增加材料数组或足以重新获取材料的稳定信息，并升级草稿版本。恢复到题组中间题时，材料卡片必须立即可见。v1 草稿继续按原逻辑加载。

## 10. 题库制作器复核页

复核页从单题列表升级为按材料折叠的层级视图：

```text
阅读材料一
  ├─ 第 1 题：采纳 / 丢弃
  ├─ 第 2 题：采纳 / 丢弃
  └─ 第 3 题：采纳 / 丢弃
```

规则：

- 用户可以查看完整材料和每道子题；
- 删除全部子题时自动删除材料；
- 保留任意子题时保留材料；
- 修改材料归属属于后续编辑能力，本阶段只支持采纳/丢弃；
- 材料或引用存在结构错误时不能进入最终打包；
- 低置信度材料边界必须明确展示原因。

## 12. 测试清单

沿用项目现有纯 `int main()` + CTest 风格。

### 校验器

- 合法的“2 份材料 + 多道子题 + 独立题”通过；
- 重复材料 ID；
- 题目引用不存在材料；
- 材料和题目分类不一致；
- 空材料；
- 孤立材料；
- 材料正文被完整复制进子题 `stem`；
- v1 题库回归通过；
- Manifest/Bank 版本不一致时失败。

### 切块和工作流

- “根据材料回答 1～5 题”不会被拆成多个独立题块；
- 同一材料和子题生成稳定 ID；
- 一轮修复能修正错误 `materialId`；
- 网络中断后恢复时不重复生成已完成材料；
- 源文件修改后拒绝复用检查点；
- 旧检查点明确拒绝，不静默迁移。

### Provider / 组卷

- v2 Bank 能 load；
- `attempt.questions` 返回去重后的材料数组和题目引用；
- 顺序模式保持题组完整；
- 随机模式不打散组内题目；
- 目标题量落在题组中间时保留完整组，并报告实际题量；
- v1 Provider 流程不回归。

### UI / 草稿

- 同组题目切换时材料持续显示；
- 切换独立题时材料区域隐藏；
- 切换另一题组时材料正确更新；
- 解析页能看到材料；
- 关闭并恢复草稿后材料仍存在；
- 复核页删除全部子题后不残留孤立材料。

### 端到端

- 准备一个确定性的 v2 fixture，至少包含两份材料、每份 3 道子题和两道独立题；
- 使用本机 `build/test-data/c3-real-reading-comprehension.txt` 做生成压测；
- 生成 `.quizpane-provider` 后通过 `ProviderInstaller::inspect()`；
- 从 ZIP 重新读取并由 `DeclarativeProvider` 加载；
- 在主程序完成进入题组、作答、交卷、查看解析和草稿恢复。

模型生成具有非确定性，CTest 不应断言 C³ 一定生成固定题数。固定数量和引用关系由手写 v2 fixture 测试，C³ 用于人工端到端与质量评估。

## 13. 推荐实施顺序

建议拆成五个可独立 Review 的变更：

1. **Schema v2 + BankValidator + fixtures**：先固定数据契约；
2. **DeclarativeProvider + 题组组卷测试**：证明产物可以运行；
3. **MainWindow 材料 UI + DraftStore**：证明用户可以正确作答和恢复；
4. **生成引擎 + 检查点 v2 + 复核页**：让制作器能够生成该产物；
5. **端到端打包与文档**：验证安装包和完整流程。

不要先改模型提示再补 Schema。没有本地确定性契约时，模型输出看似正确也无法稳定进入运行时。

## 14. 完成定义

满足以下条件才算“支持材料题”，不能只以模型成功返回 JSON 为完成：

- v2 Schema 能无重复地表达共享材料和子题；
- 本地校验器能发现所有断裂引用；
- 切块不会静默拆散明显题组；
- 检查点能恢复材料和关联题目；
- Provider 按题组组卷并返回材料；
- 主程序独立显示材料并支持草稿恢复；
- 打包后的 v2 Provider 能安装、答题和查看解析；
- v1 题库和现有测试继续通过；
- 代码和文档中不存在“复制材料到每道题 stem”的兜底逻辑。
