# 小窗刷题：题库管理与跨平台构建打包指南

更新日期：2026-07-13

## 1. 题库管理怎么使用

小窗标题栏右侧的“三横线”按钮是主菜单。菜单结构如下：

```text
题库管理
├── 添加题库…
├── 切换题库
│   └── 已安装题库列表
└── 删除题库
    └── 已安装题库列表
老板键设置…
关于小窗刷题
```

### 1.1 添加题库

选择“题库管理 → 添加题库…”，或直接将 `.quizpane-provider` 文件拖入小窗。程序会检查平台、Provider ABI、ZIP 路径安全、解压体积和动态库文件，确认后安装到当前用户目录，不要求管理员权限。

### 1.2 切换题库

选择“题库管理 → 切换题库”，再选择目标题库。程序会结束当前题库会话、加载目标题库并进入它自己的登录或分类页面。

登录状态以题库 ID 为命名空间分别保存：

- macOS：Keychain；
- Windows：Credential Manager；
- UOS/Linux：Secret Service/libsecret。

因此从一个远程题库切到本地题库，再切回原题库时，原登录态不会因为切换而丢失。草稿也按题库 ID 隔离。

### 1.3 删除题库

删除会移除题库程序文件，但默认保留安全凭据和草稿；以后重新添加同一题库时仍可恢复登录和未完成草稿。Windows 如果正在使用 DLL，程序会登记为“下次启动删除”，避免强行删除已加载模块。

## 2. 老板键管理

选择“三横线 → 老板键设置…”，在输入框中直接按下新的组合键并保存。

当前跨平台实现接受：

- 一个字母或数字主键；
- 至少一个 Ctrl、Shift、Alt 或 Meta/Win/Cmd 修饰键；
- 例如 `Ctrl+Shift+H`、`Alt+Shift+Q`。

保存时会立即尝试注册系统全局热键。若按键已被其他程序占用，会显示错误并恢复原老板键。设置保存在当前用户配置中。

## 4. 目录和产物

```text
quizpane/
├── apps/desktop-qt/                  # 通用桌面 Host
├── sdk/                              # Provider ABI、安装器和加载器
├── providers/demo/                   # 可公开的离线示例 Provider
├── tools/bank-generator/             # 本地题库生成器（开发中）
├── scripts/                          # Host 构建脚本
├── packaging/linux/                  # UOS/麒麟/Linux desktop 文件
└── dist/                             # 发布脚本输出（不提交 Git）
```

Provider 是平台相关动态库，必须分别构建：

| 系统 | 动态库 | 安装包平台键 |
|---|---|---|
| macOS Apple Silicon | `.dylib` | `macos-arm64` |
| macOS Intel | `.dylib` | `macos-x86_64` |
| Windows x64 | `.dll` | `windows-x64` |
| UOS x86_64 | `.so` | `linux-x86_64` |
| UOS ARM64 | `.so` | `linux-arm64` |
| 银河麒麟 x86_64 | `.so` | `linux-x86_64` |
| 银河麒麟 ARM64 | `.so` | `linux-arm64` |
| 银河麒麟 LoongArch64 | `.so` | `linux-loongarch64`（规划中） |

## 5. macOS 构建与打包

### 5.1 环境

```bash
xcode-select --install
brew install cmake ninja qtbase
```

### 5.2 构建 Provider

公开仓库中的 Demo Provider 会随 Host 一起编译。第三方 Provider 应在自己的仓库中使用相同架构和兼容 Qt 版本构建，并通过自身的 `package_provider` target 生成 `.quizpane-provider`。

### 5.3 构建 Host

普通开发包：

```bash
cd /path/to/quizpane
./scripts/build-macos.sh
```

Host 已显式链接 QtNetwork，因此 `macdeployqt` 会自动把在线 Provider 所需网络运行库放入 App。正式签名构建：

```bash
cd /path/to/quizpane
SIGN_IDENTITY="Developer ID Application: 你的名称 (TEAMID)" \
  ./scripts/build-macos.sh
```

没有正式证书时，默认使用 ad-hoc 签名 `-`，适合本机测试，不适合公开下载。公开发行还需：

```bash
xcrun notarytool submit dist/macos/小窗刷题-macos-arm64.zip \
  --keychain-profile YOUR_PROFILE --wait
xcrun stapler staple build/release-macos/apps/desktop-qt/小窗刷题.app
```

Apple Silicon 与 Intel 需要分别在对应架构下构建，或另建 Universal 2 工程并同时验证所有 Provider。

## 6. Windows 10/11 构建与打包

### 6.1 环境

- Visual Studio 2022 Build Tools，勾选“使用 C++ 的桌面开发”；
- CMake、Ninja；
- Qt 6.5+ MSVC 2022 x64；
- 在“x64 Native Tools Command Prompt for VS 2022”中执行。

### 6.2 Host

```powershell
cd D:\src\quizpane
.\scripts\build-windows.ps1 -QtRoot C:\Qt\6.8.0\msvc2022_64
```

脚本会运行测试、执行 `windeployqt` 并输出：

```text
dist/windows/小窗刷题-windows-x64.zip
```

### 6.3 第三方 Provider

```powershell
cd D:\src\your-provider
.\scripts\build-windows.ps1 -QtRoot C:\Qt\6.8.0\msvc2022_64
```

生成的 `.quizpane-provider` 可直接拖入 Windows 版小窗。Provider 必须与 Host 的 CPU 架构和运行库兼容。

正式发行建议用 MSIX、WiX 或 Inno Setup 包装 ZIP，并对 EXE、DLL 和安装器进行 Authenticode 签名。

> Windows 7 不属于当前 Qt 6 主线产物的验证范围，需要 Qt 5.15 兼容分支，不能把 Windows 10/11 包直接复制过去。

## 7. 统信 UOS 构建与打包

### 7.1 常用依赖

不同 UOS 版本的软件包名称可能不同，Debian 系版本可参考：

```bash
sudo apt install build-essential cmake ninja-build pkg-config \
  qt6-base-dev qt6-base-private-dev libx11-dev libsecret-1-dev
```

系统 Qt 低于项目要求时，设置 `QT_PREFIX` 指向自带 Qt：

```bash
export QT_PREFIX=/opt/Qt/6.8.0/gcc_64
```

### 7.2 Host

```bash
cd /path/to/quizpane
./scripts/build-uos.sh
```

始终生成：

```text
dist/uos/小窗刷题-uos-<arch>.tar.gz
```

若 PATH 中存在 `linuxdeploy`，脚本还会尝试生成 AppImage。UOS 对 AppImage、托盘和 X11/Wayland 的支持因版本而异，发布前必须在目标系统真机验证。

### 7.3 第三方 Provider

```bash
cd /path/to/your-provider
./scripts/build-uos.sh
```

x86_64 和 ARM64 必须分别在对应目标机或交叉编译环境构建，不能共用 `.so`。

## 8. 银河麒麟构建与打包

### 8.1 支持范围和技术路线

银河麒麟和统信 UOS 都走项目现有的 Linux/Qt/X11 代码路径，不单独维护一套 UI 源码。两者共享：

- ELF 可执行文件和 `.so` Provider；
- Qt 6 Widgets、Qt Network；
- X11 全局热键实现；
- Secret Service/libsecret 安全凭据存储；
- `.desktop` 启动项和 PNG 应用图标。

银河麒麟桌面系统存在多个版本和 CPU 架构。麒麟官方资料列出了 x86、ARM、RISC-V、LoongArch 等架构支持，但这不代表同一个二进制可以跨架构运行；本项目第一批建议只承诺经过真机测试的 **x86_64** 和 **ARM64/aarch64**。LoongArch64 需要 Qt 6、编译器和所有依赖均有对应架构版本后另行适配。[麒麟软件桌面系统架构说明](https://www.kylinos.cn/productPc/industry/industryDesktop/)

发布包必须同时标明：

```text
银河麒麟版本 + CPU 架构 + Qt 版本 + glibc 最低版本
```

不能在较新的银河麒麟环境编译后，未经验证就宣称兼容所有 V10/V11 版本。通常应在计划支持的最老系统上构建，以降低 glibc 和系统库版本要求。

### 8.2 查看目标系统信息

在麒麟真机或构建机上执行：

```bash
cat /etc/os-release
uname -m
ldd --version | head -n 1
qmake6 -v 2>/dev/null || qtpaths6 --qt-version 2>/dev/null || true
```

常见 `uname -m` 输出：

- `x86_64`：Intel、AMD、兆芯、海光等兼容架构；
- `aarch64`：飞腾、鲲鹏等 ARM64 平台；
- `loongarch64`：龙芯新世界架构。

Provider 的平台键沿用 Linux ABI 命名，而不是另建 `kylin-*`：x86_64 使用 `linux-x86_64`，ARM64 使用 `linux-arm64`。这样同一架构、依赖兼容的 Provider 可以在 UOS 和银河麒麟之间复用。

### 8.3 安装构建依赖

银河麒麟桌面 V10 的常见软件源和包管理方式与 Debian/Ubuntu 系接近，可先尝试：

```bash
sudo apt update
sudo apt install build-essential cmake ninja-build pkg-config \
  qt6-base-dev qt6-base-private-dev libx11-dev libsecret-1-dev \
  libgl1-mesa-dev
```

如果提示找不到 `qt6-base-dev`，说明当前系统源没有满足项目要求的 Qt 6。不要直接降成 Qt 5 编译，因为当前主线代码和 CMake 目标使用 Qt 6；应安装与 CPU 架构一致的 Qt 6 SDK，然后设置：

```bash
export QT_PREFIX=/opt/Qt/6.8.0/gcc_64       # x86_64 示例
# export QT_PREFIX=/opt/Qt/6.8.0/gcc_arm64  # ARM64 示例，实际目录以 SDK 为准
```

银河麒麟 V11 引入了新的系统与应用包机制，不应假设 V10 的 `.deb` 安装流程在所有 V11 环境完全一致。当前项目先交付自包含目录压缩包；进入麒麟软件商店或制作系统原生安装包时，再按照目标版本的应用规范单独制作和认证。[银河麒麟桌面操作系统 V11 产品页](https://www.kylinos.cn/productPc/desktop/desktopMainV11/)

### 8.4 构建 Host

复用 UOS/Linux 脚本，通过 `DISTRO_ID=kylin` 改变发布产物名称：

```bash
cd /path/to/quizpane
DISTRO_ID=kylin \
DIST_DIR="$PWD/dist/kylin" \
  ./scripts/build-uos.sh
```

生成：

```text
dist/kylin/小窗刷题-kylin-<arch>.tar.gz
```

例如：

```text
小窗刷题-kylin-x86_64.tar.gz
小窗刷题-kylin-aarch64.tar.gz
```

压缩包内是 `小窗刷题.AppDir`。解压后可以直接运行：

```bash
tar -xzf 小窗刷题-kylin-x86_64.tar.gz
./小窗刷题.AppDir/usr/bin/小窗刷题
```

若构建机安装了 `linuxdeploy`，脚本还会尝试生成 AppImage。麒麟部分政企环境会禁用 FUSE 或限制 AppImage，因此正式交付必须保留 `tar.gz` 目录包，不能只提供 AppImage。

### 8.5 构建第三方 Provider

Provider 继续使用 Linux 构建脚本：

```bash
cd /path/to/your-provider
./scripts/build-uos.sh
```

如果使用自带 Qt 6：

```bash
QT_PREFIX=/opt/Qt/6.8.0/gcc_64 ./scripts/build-uos.sh
```

生成的 `.quizpane-provider` 必须和 Host 使用相同 CPU 架构，并保证 Qt 主版本及 C++ 运行库兼容。不能把 x86_64 Provider 拖入 ARM64 Host，也不能把 macOS/Windows Provider 改扩展名后使用。

### 8.6 麒麟真机验收

至少在每一个声明支持的“系统版本 × CPU 架构”组合上检查：

- 应用可从文件管理器和终端启动；
- 中文字体、透明窗口、毛玻璃或降级背景正常；
- 拖动、三档尺寸和滚轮区域正常；
- X11 下老板键可注册、隐藏和恢复；
- Wayland 会话下如果全局热键不可用，应给出明确提示或建议切换 X11；
- 托盘图标在麒麟桌面环境中可见；
- PIN 置顶、菜单、题库添加/切换/删除正常；
- libsecret/系统密钥环能够保存并恢复登录态；
- 远程 Provider 能建立 TLS 连接，系统 CA 证书链有效；
- `ldd` 输出不存在 `not found`；
- 干净机器无需开发目录中的 Qt 或编译器文件即可运行；
- 退出重启后草稿和登录态仍可恢复。

建议保存以下验收信息，随发布记录归档：

```bash
cat /etc/os-release
uname -a
ldd ./小窗刷题.AppDir/usr/bin/小窗刷题
```

### 8.7 当前限制

- 当前脚本只制作自包含目录包和可选 AppImage，尚未制作麒麟软件商店包；
- LoongArch64、RISC-V 尚未完成编译和真机验证；
- Wayland 全局快捷键受桌面安全模型限制，现阶段优先保证 X11；
- 这一章节提供的是构建路径，不能在完成麒麟真机测试前宣称正式兼容。

## 9. Provider 开发者构建

第三方题库只需实现稳定 C ABI 和 JSON 协议，不需要链接 Host UI。最小构建流程：

```bash
cmake -S . -B build/release -G Ninja \
  -DCMAKE_BUILD_TYPE=Release \
  -DCMAKE_PREFIX_PATH=/path/to/Qt
cmake --build build/release --parallel
ctest --test-dir build/release --output-on-failure
cmake --build build/release --target package_provider
```

安装包应包含：

```text
manifest.json
bin/<平台动态库>
```

清单至少声明 ID、名称、版本、ABI、平台动态库和网络权限。题库 ID 一旦公开发布不要随意修改，否则登录态、草稿和升级关系都会被视为另一个题库。

## 10. 发布验收清单

- Host 和每个 Provider 的自动测试全部通过；
- 在干净系统上启动，不依赖开发机 Homebrew、Qt 或编译器目录；
- 添加、切换、删除两个以上题库；
- 需要登录的题库切走再切回仍保持登录；
- 老板键修改后立即生效，重启后仍生效；
- 关于页名称、图标、版本、作者和 License 正确；
- macOS 验证签名、公证和 Gatekeeper；
- Windows 验证签名、Defender 和 Credential Manager；
- UOS 验证 x86_64/ARM64、X11/Wayland、托盘和 libsecret；
- 银河麒麟按声明支持的版本分别验证 x86_64/ARM64、X11/Wayland、托盘、TLS 和 libsecret；
- 不在日志、安装包、崩溃报告中包含 Cookie、密码或验证码。

Windows、UOS 和银河麒麟构建入口已经提供，但本轮只在 macOS Apple Silicon 上完成实际编译和界面验收；其他平台必须在目标系统上执行脚本后才能宣称正式支持。
