# 难样本后置复核与统一 LLM 调度方案

> 背景：启发式规则能可靠处理"结构规整"的题目（题号+选项+答案都有清晰分隔符），
> 但资料分析、图形推理这类题型天然依赖图片/表格语义，规则只能校验"结构是否完整"，
> 无法校验"裁对了没有、选项对应对了没有"。现有 `needsReview` 是二元判定，只有
> 结构性缺失（缺题干/缺选项/缺答案）才会触发，语义正确性完全没有第二道防线。
> 本文档只回答三个问题：**如何发现规则做错了、如何修正、LLM 调用如何统一管理**。
> 只出方案，不动工。日期：2026-07-17。

## 0. 结论速览

1. 复核触发信号从"结构是否完整"扩展为三层：**硬失败**（现状，结构不完整必须复核）、
   **高危名单**（题型/图片依赖决定，无论规则是否报错都强制复核）、**统计异常**（把
   `scripts/audit-beijing-bank.py` 这类事后脚本的比对逻辑产品化为生成时的全局检查）。
2. 复核修正从"整题采纳/丢弃"升级为"锚点级修正"：图片裁剪框可拖拽调整、原文可划选
   重新标注题干/选项/答案边界，两者都能选择本地重算或转发 LLM 辅助。前提是先把现在
   裁完图就丢弃的裁剪矩形和文本偏移保留成可编辑数据。
3. 新增的 LLM 调用点（复核审计、建议裁剪区域、按人工指导修复单题）不再各自实现网络
   请求，统一收拢到一个 `LlmGateway`，全项目只有一处鉴权、超时、重试、Token 统计和
   诊断埋点逻辑。
4. **生成路径收敛为规则引擎单一入口**：`GenerationWorkflow` 现有的"整份文档发给模型
   结构化"路径（`start()`）退场，模型不再负责从原文生成题目结构。规则生成之后接两条
   独立、都可选的复核 lane——**人工复核**和 **AI 复核**：不配置 API Key 或无网环境下，
   用户仍能完整走完人工复核并产出题库；配置了视觉模型后，AI 复核作为审计员对规则产物
   给出确认或修正建议，不越权重新生成结构。详见第 4 节。
5. `riskLevel: soft` 允许按类别批量标记已复核（如"资料分析类全部标记已复核"），不强制
   逐题展开；`riskLevel: hard` 不允许批量操作。`suggestCrop`/`repairQuestion` 等需要
   读图的 LLM 任务要求用户已选定支持视觉输入的模型，不做纯文本降级。

## 1. 现状盘点（改动前必须确认的事实）

- `ReviewResult`（`review_result.hpp`）是规则与模型共用的唯一中间结构：
  `materials` / `questions` / `needsReviewQuestions` 三个数组，题目级别只有
  `review:{needsReview, confidence, reason}` 三个字段（`bank_validator.cpp:174-185`
  校验其结构）。
- 触发 `needsReview=true` 的条件全部是结构性硬失败（`rule_based_generator.cpp:1082-1103`）：
  缺题干、选项少于 2 个、未识别到答案、答案引用不存在的选项、多选/单选标注矛盾。
  **规则没有报错、但语义可能是错的题（资料分析算错行、图形选项裁错）会直接进
  `generatedQuestions_`，免检地进入最终包。**
- 复核页（`studio_window.cpp:buildReviewPage` + `populateReview`）是一棵 `QTreeWidget`：
  材料折叠成组，子题只能整题勾选"采纳/丢弃"，预览是纯文本 `stem.left(300)`，看不到
  原文、原图，也不能修改任何字段。
- PDF 裁图（`rule_based_generator.cpp:160-230` 材料图、`:287-425` 题干/选项视觉图）算出
  裁剪矩形后立即转成 PNG 字节写进 `assets`，矩形坐标本身不保留在任何候选 DTO 或检查点
  里，复核阶段无法"调整一下裁剪框重裁"，只能整题丢弃重扫。
- 唯一的 LLM 调用点是 `ModelClient`（`model_client.hpp/.cpp`），专为 OpenAI-compatible
  Chat Completions 整块生成设计：一次请求、一段系统提示、纯文本 JSON 响应，没有多模态、
  没有单题级任务、没有独立于生成流程的调用入口。当前它服务的是 `GenerationWorkflow::start()`
  ——把原文直接交给模型做"从零结构化"，与规则路径 `startRuleBased()` 是并列的两条独立生成
  入口，产物走同一个 `BankValidator`，但生成过程本身谁都不检查谁。**本方案第 4 节要收敛
  掉这条并列入口**，模型今后只审计规则产物，不再独立生成结构。
- 目前唯一的"全局一致性检查"是仓库外的 `scripts/audit-beijing-bank.py`：按题号连续性
  重建 PDF 基线、比对页码/选项数/答案/材料分组，但它是回归脚本，不在产品运行时里，
  普通用户拿到的制作器完全没有这类检查。

## 2. 第一步：如何发现启发式规则做错了

把触发复核的信号分成三层，层与层是"任一命中即复核"的并集关系，不是互斥分类。

### 2.1 硬失败信号（现状，保留）

结构上就不完整，规则本身承认解析失败：

- 缺题干 / 选项少于 2 个 / 未识别到答案 / 答案指不存在的选项；
- 多选题字段与识别到的答案数矛盾（题干标多选但只有一个答案，或反之）。

这一层不需要改动，继续无条件复核。

### 2.2 高危名单信号（新增，对应用户提出的"资料分析、图形推理类必须复查"）

这一层的特点是：**规则没有报任何错，题目结构完整，但内容正确性规则原理上验证不了**。
判定不依赖"是否命中某条报错规则"，而依赖题目本身的类型特征，命中即强制进复核队列，
哪怕当前 `confidence` 很高：

| 特征 | 判定方式 | 典型场景 |
|---|---|---|
| 资料分析 | 关联材料的 section 标题命中"资料分析"（复用 `rule_based_generator.cpp:83` 已有的段落识别）、或材料 `images` 非空且 `chartOnly` | 统计图表读数、算比例 |
| 图形推理 | section 标题命中"图形推理"、或选项本身是图片（`optionImages` 非空） | 图形找规律 |
| 类比推理、定义判断中的图示题 | 题干或选项包含图片资源 | 少见但存在 |
| 跨页题 | `source.page` 与相邻题不连续，或题干裁剪跨越两个 `pageImages` | PDF 排版跨页 |
| OCR 来源 | `document.usedOcr == true`（已用于 confidence 0.75 vs 0.95，需要提升为强制复核而非只降置信度） | 扫描件 |

这一层不阻止题目"可用"，但要求它在复核页默认展示、默认不隐藏为"已通过"，需要用户至少
看到分类摘要（如"资料分析类 12 题"）并主动确认（逐题或按类批量，见 2.4）才放行，因为
规则对这类题的正确性判断能力约等于零。

### 2.3 全局/统计级信号（新增，把回归脚本的逻辑产品化）

单题看不出问题，只有跟"这批题应该是什么样子"做比对才能发现，本质就是把
`audit-beijing-bank.py` 里已经验证有效的检查搬进生成流程里，实时跑：

1. **题号连续性**：源文档能提取出连续题号序列（`\d+[、.．]` 开头）时，检测生成结果
   是否覆盖了从最小到最大的所有题号，缺口列出具体缺失的题号，而不是只报"总数不对"；
2. **页面覆盖率**：文字型 PDF 每一页至少应关联到一道题或一份材料的 `source.page`；
   完全没有被引用的页面提示"可能整页被跳过"；
3. **材料分组边界**：同一份材料关联的题目题号应当连续（`audit-beijing-bank.py:109-113`
   的"两个独立资料组被合并"检查）；相邻材料的 `materialId` 集合不应完全相同；
4. **选项数量分布**：同一份文档里选项数应该基本一致（如清一色 4 选项），个别题目选项
   数偏离众数时标记，可能是选项被截断或多算了一行；
5. **答案分布**：单一答案字母占比异常高（如超过 60% 都是同一个字母）时标记，这通常
   指向选项/答案对齐系统性错位，而不是巧合；
6. **题干长度异常**：显著短于同批次中位数（可能题干被截断）或显著长于中位数（可能
   材料未拆分、多题被误合并成一题）时标记；
7. **裁剪图尺寸/内容异常**：裁出来的图片长宽比超出合理范围、或近似纯色（复用
   `image_privacy_filter` 已有的像素扫描能力做"是否几乎全白/全黑"检测），提示裁剪框
   可能落空；
8. **独立答案文档比对**（可选，仅当用户配对了单独的答案/解析文件时）：像
   `pairAnswer` 已经支持的那样，独立提取答案文档里的题号-答案对，与规则从题目文档里
   解析出的答案比对，不一致时标记 `answer-mismatch`，这是当前信号里唯一能真正验证
   "答案对不对"而不只是"结构全不全"的手段，应当优先在有配对答案文件时启用。

这一层的检查对象是"一批题目"而不是单题，因此不适合放在
`rule_based_generator.cpp` 单题解析函数内部，需要在规则生成器跑完整份文档后，
新增一个独立的汇总检查阶段（建议命名 `RuleBasedGenerationAudit`），输出的问题以
`{questionIds/materialIds, kind, detail}` 形式挂回对应题目的 `review.signals`，
而不是笼统地把整批打回复核。

### 2.4 复核信号的统一表达

现有 `review:{needsReview, confidence, reason}` 是二元的，无法表达"结构合法、但属于
高危类型/统计异常"这种"不算错但要看一眼"的中间态。建议扩展为：

```json
"review": {
  "needsReview": true,
  "confidence": 0.6,
  "reason": "资料分析题：规则无法验证图表读数正确性",
  "riskLevel": "soft",
  "signals": ["material-type:资料分析", "ocr-source"]
}
```

- `riskLevel: "hard"`：结构性缺失，沿用现状，禁止批量通过，必须逐题处理；
- `riskLevel: "soft"`：结构合法但命中高危名单或统计异常，默认展示在复核页，**允许按
  `signals` 分类批量标记已复核**（如复核页点一次"资料分析类全部标记已复核"，把该分类
  下所有 soft 题的勾选状态从"未确认"改成"通过"），但默认状态仍是未确认、不隐藏、不静默
  进最终包；
- 不带 `riskLevel` 或省略：完全常规题，维持现状直接进最终包，不出现在复核页。

批量操作的语义是"用户对这一类做了抽查、愿意整体放行"，不是自动豁免——批量按钮本身要求
用户先点开看过复核页里这一类的样例，而不是无脑一键跳过全部风险题（这属于复核页交互细节，
本文档不展开具体交互形态，只锁定数据层允许分类批量确认）。

`bank_validator.cpp:174-185` 的 `reviewKeys` 白名单需要相应扩展 `riskLevel` 和
`signals`，`signals` 校验为字符串数组即可，不需要枚举穷举，避免每加一种信号就要改
Schema。

## 3. 第二步：修正方式设计

用户的两个具体诉求：**图片锚点调整**（裁剪框拖对地方）、**划文字标注**（原文里重新框
定题干/选项/答案边界），并且都要能选择人工直接做或者请 LLM 按指导重新生成。

### 3.1 前提：锚点必须先变成可编辑数据

当前裁剪矩形（`rule_based_generator.cpp` 里一堆 `qFloor(source.width() * top)` 这类
计算）算完立刻转 PNG 丢进 `assets`，候选 DTO 和检查点里完全不保留矩形坐标本身。不解决
这个问题，"调整裁剪框重裁"就无从谈起，只能"整题丢弃、回去改源文件重新扫一遍"，这正是
现在体验差的根源。

需要新增一个只在复核阶段存活、不写入最终 `bank.json` 的锚点结构：

```cpp
// 仅用于题库制作器复核阶段；确认后落地成最终 stemImage/材料 body，不进 bank.json。
struct ReviewAnchor {
    QString targetId;      // question.id 或 material.id
    QString sourcePath;
    int page = 0;
    // 图片类锚点：归一化 0..1 的裁剪框，直接对应现有 crop 计算里已经算出的比例。
    std::optional<QRectF> imageCrop;
    // 文本类锚点：该 page plainText 内的字符偏移，标注题干/选项/答案/解析分别对应哪段原文。
    QHash<QString, std::pair<int,int>> textSpans; // key: "stem"/"option:a"/"answer"/"solution"
};
```

`ReviewResult` 增加 `QJsonArray reviewAnchors`，`CheckpointStore` 同步保存，使得暂停/
恢复后复核页仍能展示可编辑锚点，而不是只剩最终裁好的 PNG。

### 3.2 交互一：图片锚点调整

面向资料分析图表、图形推理选项这类"裁剪框位置决定对错"的题：

- 复核页对命中"高危名单"或裁剪图异常信号的题，展示该题来源整页缩略图，当前裁剪框以
  半透明遮罩 + 高亮边框叠加显示（类似系统截图工具的选区）；
- 支持拖拽四角/四边调整选区，也支持清空后重新拉一个矩形；
- "本地重新裁剪"按钮：直接用调整后的 `QRectF` 对 `document.pageImages` 重新执行现有
  裁剪+透明化流程，纯本地位图操作，即时出结果，不需要 LLM；
- "AI 建议裁剪区域"按钮（可选）：把整页图、当前裁剪框、问题描述（如"这题应该有 4 个
  选项但只裁到 3 个"）一起发给支持视觉输入的模型，模型只返回建议的归一化矩形，本地
  校验合法性（在页面范围内、面积不为零）后回填到可调整选区，用户仍需要拖动确认或直接
  接受，不允许模型结果免检落地。

### 3.3 交互二：文字划选标注

面向"规则切错边界"的题（题干被截断、选项黏连、答案区域跑偏），这类问题图片裁剪解决
不了，需要重新圈定原文范围：

- 复核页展示该题来源原文的一段 `plainText`（按 page/line 定位，前后留出几行上下文），
  用户在原文里拖拽选中一段连续文本；
- 选中后弹出语义槽位选择（题干 / 选项 A / 选项 B / … / 答案 / 解析），把这段文本偏移
  绑定到 `ReviewAnchor.textSpans` 对应的 key；
- 提交后跑一次轻量转换（不是重新调用整个规则引擎，只是"这段标注文字就是这个字段的
  值"），再过一次 `BankValidator`，不合法的部分（如答案标注的文字不在任何选项里）立即
  提示，不静默接受；
- 这个交互解决的是"人工比规则更快认出边界在哪，但不想手输 JSON"的场景，比强迫用户在
  文本框里逐字重打题干/选项友好得多。

### 3.4 交互三：直接文本编辑（兜底）

对只是错别字、答案标注错误这类不需要重新定位边界的场景，保留一个简单表单直接编辑
题干/选项/答案文本，作为前两种交互的降级方案。这条路径现在其实已经隐含存在（复核页
预览可以扩展成可编辑），改动量最小，优先级最低但不可省略——不是所有错误都值得拖动作。

### 3.5 人工路径与 LLM 辅助路径共享同一个修正意图

无论用户是拖裁剪框、划文字，还是直接编辑文本，最终都产出同一种"修正意图"对象：

```cpp
struct RepairIntent {
    QString targetId;
    QJsonObject anchorBefore;   // 调整前的锚点/字段快照，用于对比和撤销
    QJsonObject anchorAfter;    // 用户确认的锚点/字段
    QString userHint;           // 可选：给 LLM 的自然语言指导，如"选项D在下一页顶部"
};
```

- **人工确认**：直接用 `anchorAfter` 本地重算（重裁图或重赋值文本字段），不调用模型；
- **LLM 辅助**：把 `RepairIntent` 连同原文块、当前题目 JSON、之前的复核 `reason` 一起
  发给 `LlmGateway`（见第 4 节），模型只返回该题的修正 JSON，复用现有修复循环同一套
  `BankValidator` 校验，通过才回填，不通过维持复核状态、保留错误详情。

两条路径共享同一份校验和回填代码，区别只在于"最终内容谁提供"，避免维护两套语义不同的
"确认题目"逻辑。

## 4. 第三步：LLM 调用统一收拢，兼并为"规则生成 + AI 复核"架构

### 4.1 架构变化：AI 不再独立生成，只审计规则产物

现状 `GenerationWorkflow` 有两条并列生成入口：`startRuleBased()`（规则）和 `start()`
（整份原文交给模型从零结构化）。两条路径互不知情，产物才在 `BankValidator` 汇合。

**本方案收敛为规则引擎单一生成入口**：所有题库都先经过 `RuleBasedBankGenerator` 产出
`ReviewResult`，`GenerationWorkflow::start()`（整块模型生成）连同 `kSystemPrompt`、
`bankForCandidate` 里"模型直接产出 materials+questions"的语义一并移除。原因：

- 规则路径是确定性的、可复现、不依赖网络和 API Key，`doc/2011北京卷PDF回归报告.md`
  证明它已经能把结构规整的题目做到 135/135 全对；模型独立生成反而引入了规则路径不存在
  的不确定性（幻觉、格式漂移），却没有换来更好的结构可靠性；
- 两条路径分别校验、分别修复，维护成本是双份的，而用户真正需要的是"规则搭好骨架，AI
  帮忙挑错"，不是"两套引擎抢着生成、比谁的结果能通过校验"。

规则生成完成后，进入两条**都是可选的、互不依赖**的复核 lane：

```text
                                ┌─ 人工复核（锚点交互，见第 3 节，纯本地，无需网络） ─┐
RuleBasedBankGenerator ─► ReviewResult ─┤                                          ├─► 最终 bank.json
                                └─ AI 复核（LlmGateway 审计 + 建议，需配置模型） ────┘
```

- 未配置 API Key 或无网环境：AI 复核入口置灰，用户仍可完整走人工复核产出题库，这是
  必须始终成立的底线路径；
- 配置了支持视觉输入的模型：AI 复核作为"审计员"介入，对 `riskLevel: soft` 的题给出
  "同意 / 建议修正"的判断，对用户发起的锚点调整给出修正建议，但**不产出新题、不越权
  改动规则没有标记为待复核的题**，所有输出仍要过 `BankValidator` 且需要用户确认才落地
  （维持第 3.5 节"不允许模型结果免检落地"的约束）。

### 4.2 为什么仍需要收拢

即便去掉了整块生成任务，复核阶段新增的调用点（AI 复核审计、建议裁剪区域、按人工指导
修复单题）如果各自新写一遍网络请求，仍然会重复实现：厂商 Endpoint 差异、鉴权头、
超时/取消、Token 统计、诊断日志埋点、错误信息归一化。这些逻辑已经在 `model_client.cpp`
里写过一次，收拢进统一入口是为了不写第二遍，不是为了保留"整块生成"这个任务本身。

### 4.3 定位

新增 `LlmGateway`（建议路径 `apps/bank-studio/engine/include/quizpane/studio/llm_gateway.hpp`
+ 对应 `.cpp`），作为进程内唯一的模型调用入口。现有 `ModelClient` 降级为它内部的传输
细节（HTTP 请求构造/解析），`GenerationWorkflow` 不再直接持有 `ModelClient`。

对外只暴露"任务"概念，不是"发一条 chat completion"，且所有任务都以规则产物为输入，
不再有"从原文生成"这一种任务：

| 任务 | 输入 | 输出 | 使用方 |
|---|---|---|---|
| `auditQuestion` | 规则产出的题目 JSON + 原文块（+ 相关页面图，若模型支持视觉） | 同意 / 建议修正 JSON + 理由 | AI 复核 lane（新增，替代原 `structureDocument`） |
| `suggestCrop` | 页面图、当前裁剪框、问题描述 | 建议的归一化矩形 | 复核页图片锚点交互（新增） |
| `repairQuestion` | 当前题目 JSON、原文块、`RepairIntent` | 单题修正 JSON | 复核页文字划选/LLM 辅助交互（新增） |

三个任务都要求用户已选定**支持视觉输入**的模型（对应你在开放问题 3 的回答：不做纯文本
降级）；模型目录里不支持视觉输入的模型，AI 复核入口和上述按钮直接置灰并提示原因，不发出
必然失败或质量打折的请求。

所有任务共用同一套：鉴权头构造、Endpoint 策略、超时/取消/重试、Token 用量统计、
`diagnostic_logger` 埋点、请求/响应体截断日志。新增任务类型只需要实现"构造 body / 解析
response"这一小段，不需要重新处理网络层。

### 4.4 会话与并发语义

AI 复核阶段的调用都是"用户触发或后台批量审计的小任务"（一题一次，可能是用户点开某道
soft 风险题时触发，也可能是复核页打开后台自动对所有 soft 题跑一轮审计）。`LlmGateway`
内部需要用任务队列 + 可插拔任务类型来处理，而不是让 `StudioWindow` 各处自行
`new ModelClient`。批量审计与用户手动触发的单题任务共用同一个队列，允许用户手动任务
插队到批量审计前面（用户当前正在看的题应该优先出结果）。

### 4.5 多模态支持

`auditQuestion`/`suggestCrop` 都要求把 PNG 编码成 base64 传给支持视觉输入的模型，这是
现有 `buildChatCompletionsBody` 完全没有覆盖的能力。需要：

- 模型目录（`docs/题库生成器UI与集成方案.md` 里已经列出的厂商列表）标注每个模型是否
  支持图片输入；
- 用户选择的模型不支持视觉输入时，AI 复核整体入口（不只是"建议裁剪"按钮）直接置灰并
  提示原因；
- OpenAI-compatible 的 `content` 字段需要扩展成数组形式支持 `image_url`，其他厂商
  （Anthropic Messages 等）的图片编码格式不同，这部分差异收敛在 `LlmGateway` 内部，
  调用方只传 `QImage`/`QByteArray`，不关心具体厂商怎么编码。

### 4.6 安全与隐私边界

延续 `docs/AI题库生成与声明式Provider方案.md` 第 8 节已经定下的原则：不擅自上传用户
未授权的原文。AI 复核审计的输入范围严格限定为"规则已经切出来的这一道题及其直接原文
块"，不会像旧的整块生成路径那样把整份文档主动发送出去——这也是收敛掉 `start()` 路径
带来的附带好处，默认发送的数据量比原来小得多。复核阶段的划选/裁剪同理，只发送用户
明确圈定的片段。API Key/Endpoint 复用现有 `ModelSettings`，不为复核阶段单独收集一套
凭据。

## 6. 落地顺序建议

1. **触发信号先行，不改交互**：把高危名单判定和全局统计检查产品化，输出
   `riskLevel`/`signals`，复核页仍用现在的树形列表展示，只是覆盖面变广。这一步能立刻
   收窄"规则误判但静默通过"的敞口，且不依赖任何 UI 改动，也不依赖后面几步。
2. **下线并列的模型生成入口**：移除 `GenerationWorkflow::start()`，`startRuleBased()`
   成为唯一生成入口。这一步是纯删除 + 收口，风险低，且让后续"AI 复核"从一开始就是
   附加在规则产物之上，不会跟旧的"模型生成"语义混淆。
3. **锚点数据结构先行，不改交互**：`ReviewAnchor` + `reviewAnchors` 落进检查点，确保
   裁剪矩形和文本偏移不再随裁图完成即丢弃，为后续交互打好数据基础。
4. **LlmGateway 收拢**：新建 `LlmGateway`，`ModelClient` 降级为内部传输实现；此时先只
   支撑"手动触发单题审计"这一种任务，验证鉴权/超时/取消/Token 统计不回归。
5. **图片锚点调整交互**：拖拽裁剪框 + 本地重裁，先做本地路径，验证可用后再接
   `suggestCrop`。
6. **文字划选标注交互**：原文划选 + 轻量转换 + 校验回填。
7. **AI 复核批量审计接入**：`auditQuestion` 任务类型 + 复核页"按类批量标记已复核"入口
   （对应第 2.4 节 soft 批量确认）；`repairQuestion` 任务类型接入人工划选/框选基础上的
   "求助 AI"入口。
8. **回归验证**：用 2011 北京卷等已有真题基线（`doc/2011北京卷PDF回归报告.md`）覆盖
   新复核流程，确认新增信号没有引入新的漏题、错位或误报洪水；额外验证"未配置 API Key /
   断网"场景下人工复核路径完整可用，不因 AI 复核入口置灰而阻塞任何操作。

## 7. 明确不做的事（避免过度设计）

- 不做"LLM 自动判定 riskLevel"取代规则统计信号——规则/统计信号必须先行且是确定性的，
  LLM 只用于生成"建议"，最终仍必须过 `BankValidator`，不允许模型自证正确；
- 不做通用 OCR 后自动纠错，复核阶段的人工确认步骤不可省略；
- 不保留"模型独立生成题目结构"的入口，AI 只审计规则产物，不再存在"规则生成"和"模型
  生成"两条平行、互不知情的生成路径；
- 不为纯文本模型做视觉任务的降级适配，`auditQuestion`/`suggestCrop`/`repairQuestion`
  统一要求用户已选定支持视觉输入的模型；
- 图片锚点调整、文字划选标注只服务复核阶段的单题修正，不做成通用图片标注/文本标注
  工具产品化。

## 8. 已确认的决策

1. `riskLevel: soft` 允许按分类批量标记已复核（如"资料分析类全部标记已复核"），复核页
   需要提供分类批量操作入口，不强制逐题展开。
2. 生成路径收敛为规则引擎单一入口；AI 生成路径退化为规则生成之后的 AI 复核 lane。规则
   生成之后接两条独立、都可选的复核 lane——人工复核和 AI 复核；未配置 API Key 或无网
   环境下，人工复核路径必须完整可用，AI 复核入口置灰但不阻塞产出题库。
3. `auditQuestion`/`suggestCrop`/`repairQuestion` 均要求用户已选定支持视觉输入的模型，
   不做纯文本降级；模型不支持视觉输入时相关入口直接置灰并提示原因。
