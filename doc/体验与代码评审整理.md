# QuizPane 体验与代码评审整理

> 视角：① 以「在职备考用户」的身份审视能用、好用、想用但还没有的功能与 UI/交互；
> ② 以「code reviewer」的身份审视代码效率、风格、模块拆分、注释、打包速度。
> 下列条目均为建议，标注了文件:行号，可酌情取舍。日期：2026-07-15。

---

## 第一部分：面向普通用户（在职备考）

### A. 用起来比较难受的交互 / 体验痛点

1. **做完一题会被强制 120ms 后跳下一题，打断“改主意”节奏**
   `apps/desktop-qt/src/ui/main_window.cpp:811-821` —— 选完答案后 `chooseAnswer` 用
   `QTimer::singleShot(120ms)` 自动推进。在职备考常会先点一个答案、再想一下要不要改，
   120ms 几乎是“一眨眼就翻篇”，手慢的用户根本来不及回退到上一题核对。
   建议：把自动推进做成可开关的偏好（默认开但可关），并延长到 600–1000ms；
   最后一题“答完即弹交卷确认”对想再检查的用户也略激进，建议改为“答完不自动交卷”。

2. **没有“上一题/下一题”的键盘快捷键，只能鼠标点按钮**
   `main_window.cpp:280-284` 的 prev/next 只绑定了按钮 clicked。小窗场景下键盘流（方向键、
   `J/K`、`1-4` 选 ABCD）更顺手，但现在只能用鼠标。对“空档摸鱼”的用户，
   双手不离键盘做几题体验会好很多。
   建议：加 1–9 数字键直接选选项、`←/→` 或 `J/K` 翻题、`Enter` 交卷。

3. **选项只能单选，且明确不支持多选，多选题会被判为“待复核”丢弃**
   `apps/bank-studio/engine/src/rule_based_generator.cpp:578-579`：
   “识别为多选题，但当前答题运行时暂不支持多选”。很多考试（如考研政治、行测常识）
   都有多选题，对备考用户来说这是硬缺口——导入了带多选题的资料，结果题目被静默丢弃。
   建议：答题运行时支持多选；至少在解析页提示“本题识别为多选，已跳过”，别让用户
   以为题库做错了。

4. **老板键只接受“修饰键 + 字母/数字”，且在 Linux Wayland 下不可用**
   `apps/desktop-qt/src/platform/global_hotkey.cpp:24-37` 限定单键组合；
   `:137-177` 只实现了 macOS/Win/X11，Wayland 直接 fallback 到“当前平台尚不支持”
   （TODO.md 也列为 P2）。对 UOS/麒麟这类默认 Wayland 的国产系统，老板键等于失效，
   而 README 把这些系统列为支持目标。
   建议：Wayland 用 `org.gnome.Shell`/`kglobalaccel` 或 portal DBus 方案；至少在
   不可用时给一个“应用内快捷键（仅窗口在前台时生效）”的降级提示。

5. **草稿恢复是“一次性弹窗”，点否就丢，没有“稍后”**
   `main_window.cpp:585-587`：发现草稿直接 `QMessageBox::question` 问是否恢复，
   选“否”即放弃。在职用户可能只是临时打开看看，没打算继续这套题，但也不希望直接丢。
   且草稿是按 providerId 单份保存（`draft_store.cpp:20-30`），开新练习会
   `draftStore_.clear()`（`main_window.cpp:710`）覆盖旧草稿，等于“一套题只能存一份进度”。
   建议：草稿改成按 attemptId 多份保存；恢复对话框加“稍后再问”。

6. **交卷后只能逐题线性看解析，没有“只看错题/收藏错题”**
   `main_window.cpp:901-938` 的解析页是 prev/next 线性浏览，无错题过滤、无标记收藏、
   无“导出错题本”。对备考用户，错题回顾是核心诉求，目前只做了“看一次解析”。
   建议：解析页加“仅错题”筛选、错题收藏、错题本导出（复用 `.quizpane-provider` 打包）。

7. **分类页每条只给 2 个题量按钮（5/15），且取 `counts.mid(0,2)`**
   `main_window.cpp:694`：`for (const int count : counts.mid(0, 2))` 硬截断成 2 个。
   用户想要“10 题”“30 题”“全部”都得迁就预设值；想做的题量和实际可练不匹配时无从选择。
   建议：放一个“自定义题量”输入或下拉，5/10/15/20/全部 是常见档位。

8. **题库管理藏在三级菜单里，切换/删除要进“主菜单 → 题库管理 → 切换/删除”子菜单**
   `main_window.cpp:1288-1350`。题库一多，每次切换都要点三次、再扫列表。
   对“换一套题或换个方向”这种高频操作路径偏深。
   建议：做一个题库管理页（左侧列表 + 切换/删除/导出），替代多级子菜单。

9. **删除题库“重启后删除”的提示对用户不透明**
   `main_window.cpp:1430-1437`：删不掉就塞进 `providers/pendingDelete`，
   下次启动 `processPendingProviderDeletions()` 重试，但只提示“将在重启后删除”。
   用户不知道要重启哪个、是不是真删了。README 也承认“彻底清除数据的入口还在完善”。
   建议：明确提示原因（文件被占用？）和触发时机，并给一个“强制清理”入口。

10. **无深色/浅色主题切换，全程强制深色半透明**
    `main_window.cpp:1626-1775` 的 QSS 写死深色。在某些浅色壁纸/强光环境下，
    半透明深色卡片对比度不足，看久会累；部分文字色（如 `#9ca3ab`）在浅背景上更糊。
    建议：至少支持跟随系统主题，或提供浅色变体。

### B. UI 不好看 / 不一致的地方

11. **小窗（desktop-qt）和制作器（bank-studio）是两套完全不同的视觉语言**
    `main_window.cpp` 用 `rgba(12,14,18,96)` 暗灰暖色卡片 + 圆角小按钮；
    `studio_window.cpp:802-876` 用 `#14182a→#0a0c16` 蓝紫渐变 + 蓝色主按钮。
    同一产品两个 App 风格割裂，用户从制作器回到小窗会有“换了个软件”的错乱感。
    建议：统一一套 design token（主色、圆角、间距、字号阶梯），两边复用。

12. **主界面字号靠 `[uiSize="small/medium/large"]` 属性选择器逐条重复**
    `main_window.cpp:1633-1705` 里几乎每个文本类对象都有三份 `[uiSize=…]` 字号规则，
    大段重复，改一个字号要改十几处。维护性差、也容易不一致。
    建议：用 `font-size` 跟随一个根字号变量（如设置 card 的基础 font-size，
    子元素用 `em`/`rem` 等价写法），而不是三选一硬编码。

13. **手绘图标（line_icons.cpp）线条过细且无高 DPI 适配**
    `apps/desktop-qt/src/ui/line_icons.cpp:13-17`：24×24 固定 QPixmap，1.5px 线宽，
    没有 `setDevicePixelRatio`。在 Retina/高缩放下会发虚，按钮图标偏小偏暗。
    建议：用 SVG 或按 `devicePixelRatio` 放大 pixmap，或改 `QIcon` + 多分辨率。

14. **目录行 `catalogRow` 把题量按钮挤在右侧，长题名换行后排版乱**
    `main_window.cpp:677-702`：题名 `QLabel` 是 `wordWrap`，配两个右侧按钮，
    题名长时按钮会上下错位。三档尺寸下小窗更窄，视觉更挤。
    建议：题量按钮放题名下方一行，或用横向流式排布。

15. **解析页“你的答案 / 正确答案 / ✓✗”挤在一行，长选项无换行控制**
    `main_window.cpp:925-927`：三个信息用全角空格拼接成一个字符串，
    小窗窄时会折行、`✓/✗` 与文字距离不稳定。
    建议：拆成结构化小块（你的答案、正确答案、对错徽章），用布局而不是空格对齐。

### C. 还没有做、但作为备考用户想要的功能

16. **无统计/进度仪表盘**：做了多少题、正确率、连续打卡天数、薄弱知识点分布——
    备考最看重“量变到质变”的反馈，目前完全没有。草稿只存当前未完成的一套。
    建议：本地轻量统计库（按 providerId/分类），一张“本周练习”卡片。

17. **无定时/番茄式提醒**：“空档做两道”是产品定位，但没有“每空闲 30 分钟提醒我
    做 3 题”的入口。老板键只能隐藏，不能“到点叫我”。
    建议：可选的轻提醒（尊重老板键优先级）。

18. **无多选、判断题以外的题型**（填空、简答、配对），见 A.3。
    当前 schema 也只认 `single_choice`/`true_false`（`bank_validator.cpp:16-19`）。

19. **无题库市场 / 一键找题库**：README 列为“正在完善”，目前只能自己导入或做。
    对非技术用户，拿到 `.quizpane-provider` 文件本身就是门槛。

20. **无导入/导出个人做题记录**：换机、分享给同学都做不到。

21. **小窗无法固定在某一题“挂读”**：图钉只置顶窗口，没有“停在某题反复看解析”的
    收藏态。结合 C.16 错题本可解。

### D. 影响用户的健壮性 / 边界缺口

22. **网络错误靠字符串 `contains` 匹配英文关键词**
    `main_window.cpp:89-100`：用 `contains("timed out")/("host not found")` 翻译
    Qt 网络错误。错误信息一旦本地化或措辞变化，用户就看到英文堆栈式报错。
    建议：用 `QNetworkReply::NetworkError` 枚举分支，不要靠字符串匹配。

23. **声明式题库 `catalog.list` 每次都 O(题×分类) 计数**
    `sdk/src/declarative_provider.cpp:174-175`：每个 catalog 都遍历全部 questions
    统计可练数。题库大、分类多时目录加载会卡。建议：load 时预算一次分类→题数索引。

24. **二维码绘制用 `setPixelColor` 逐像素四重循环**
    `main_window.cpp:952-958`：对大二维码是 O(n²×scale²)，且 `setPixelColor` 极慢。
    生成稍慢会卡 UI 线程（虽是登录态低频，仍属可优化）。建议：用 `QImage::scanLine`
    批量填行，或直接构造黑白 1-bit 图。

25. **解析里 `optionsHtml` 用 `toHtmlEscaped` 后再用 `plainText` 再转义**
    `main_window.cpp:916-922`：先 `plainText()`（去 HTML）再 `toHtmlEscaped()`，
    逻辑上能跑但绕；题干用 RichText、选项又转纯文本，渲染风格不一致
    （题干可带格式，选项永远是纯文本）。

26. **OCR 失败直接整份 PDF 报错返回**，不跳过坏页继续。
    `document_extractor.cpp:291-303`：某一页 OCR 失败 `result.error = …; return result;`
    导致整份资料不可用。建议：跳过该页、汇总失败页码，让用户至少能用部分内容。

## 第二部分：面向开发者（code reviewer）

### A. 效率 / 打包速度

27. **未启用预编译头（PCH）与 Unity Build，C++20 + Qt 模块全量编译慢**
   根 `CMakeLists.txt`、各子 `CMakeLists.txt` 均无 `target_precompile_headers` /
   `CMAKE_UNITY_BUILD`。Qt Widgets 项目头文件重（`<QMainWindow>`、`<Qt6/Network>` 等）
   每个 cpp 都重复解析。开发态增量构建体验差。
   建议：为 `quizpane_core`、`question_maker_engine`、两个可执行文件加一个
   `cmake_pch.hxx`（含常用 Qt 头 + 项目公共头），或开 `UNITY_BUILD_BATCH_SIZE`。

28. **`FetchContent` 在 configure 阶段联网拉 `qrcodegen` / `miniz`，无锁定**
   `CMakeLists.txt` 的 `FetchContent_Declare(... GIT_SHALLOW TRUE)`。
   首次配置需联网、CI 可能被墙；`GIT_SHALLOW` 也无法保证可复现（tag 可能被移动）。
   建议：用 `FetchContent_Declare(... SOURCE_DIR ...)` 指向 vendored 第三方源码目录，
   或 `GIT_TAG` + 锁定 commit hash，并在 README 标注离线构建方式。

29. **没有 `ninja` 之外的 ccache/编译缓存意识，CI 全量重编**
   `CMakePresets.json` 只设了 generator=Ninja。建议显式检测 `ccache`/`sccache`
   并 `CMAKE_CXX_COMPILER_LAUNCHER`，CI 与本地都受益。

30. **`declarative_provider.cpp` 的 `buildUnits` / `catalog.list` 多处 O(n²)**
   `declarative_provider.cpp:174` 每分类线性扫全量题；`:99-116` `buildUnits` 用
   `QHash` 但 `units[it.value()].append` + 重新赋值整段，题组多时有拷贝开销。
   见 D 侧用户影响（第 23 条）。建议预算索引、`QJsonArray` 引用就地修改。

31. **`rule_based_generator.cpp` 里大量 `static const QRegularExpression` 局部函数级**
   函数内 `static const QRegularExpression` 在 C++20 是线程安全首次初始化，没问题；
   但 `globalAnswers` 等函数里同一行多次 `QRegularExpression(QStringLiteral(...))`
   构造（如 `:210-211` `a.contains(QRegularExpression(...))`）是每次调用都编译正则。
   热路径逐行编译正则开销不小。建议：把这些临时正则也提到 `static const`。

### B. 代码风格 / 注释

32. **`main_window.cpp` 单文件 1778 行，一个类承担：窗口、路由、状态机、样式、拖拽、
    托盘、老板键装配、题库管理**。头文件 `main_window.hpp` 成员近 60 个裸指针。
    虽然注释说明了 Qt 父子对象树所有权（这点很好），但职责远超一个 Window。
    建议：拆出 `ProviderResponseRouter`、`CatalogPage`、`PracticePage`、`SolutionPage`、
    `DesktopShell`（托盘+菜单+热键）几个类；样式 QSS 移到 `.qss` 资源文件。

33. **QSS 内联在 C++ 原始字符串字面量里**（`main_window.cpp:1626-1775`、
    `studio_window.cpp:802-876`），近 150 行样式不可热调、无语法高亮、不能复用。
    建议：移到 `resources/*.qss` 经 `.qrc` 打包，运行时 `setStyleSheet` 读取。

34. **注释风格整体优秀，但存在“过度类比”注释**
    多处把 Qt 概念类比 jQuery/Java/Maven（如 `main_window.cpp:107-109`、
    `provider_loader.cpp:51-52`、`global_hotkey.cpp:91`、`bank_validator.cpp` 附近）。
    对资深 C++ 读者略冗余；对新手友好。可保留，但建议收敛到项目导读文档，
    代码内只留与该处逻辑强相关的“为什么”。

35. **中英混用不一致**：部分文件注释全中文（`provider_loader.cpp` 的中文注释很好），
    部分 `global_hotkey.cpp:86`、`:183` 是英文，`document_extractor.cpp:36` 中文
    紧邻英文标识符。建议统一中文为主、英文仅用于标准库/Qt API 说明。

36. **`bank_validator.cpp` 的超长 `if` 条件链**
    `bank_validator.cpp:59-67`、`:233-243` 单个 `if(... || ... || ...)` 跨十几行，
    把“标识重复 / 标题空 / 模式无效 / 计数越界”全归成一条“分类…无效”错误信息，
    定位困难。建议：拆成多条细校验，分别 append 不同 message，便于生成与修复。

### C. 模块拆分 / 耦合 / API 卫生

37. **`ProviderLoader` 把平台凭据读写（Keychain/CredManager/libsecret）全塞进
    `secureReadThunk/Write/Delete` 巨型 `#if` 分支**（`provider_loader.cpp:227-426`），
    单文件 300+ 行平台特定代码。这与“loader 负责 ABI/调度”的职责混在一起。
    建议：抽 `SecretStore` 独立类/文件，loader 只持有引用；便于单独测试与移植。

38. **`MainWindow` 直接依赖 `ProviderInstaller`、`DraftStore`、`ProviderLoader` 三个
    服务对象为值成员**（`main_window.hpp:105-107`），并在 UI 里直接调用文件路径、
    ZIP 校验、凭据逻辑，UI 与持久化/安装层强耦合。
    建议：引入一个薄 `AppServices` 接口，UI 只依赖接口，便于替换与测试。

39. **`generation_workflow.cpp` 把模型校验失败时“整批降级为待复核”的补救逻辑
    写死在 workflow 里**（`:362-381`），与 `rule_based_generator` 的复核语义重复。
    两条生成路径（规则 vs 模型）对“复核”的产出结构不一致，后续维护要两处改。
    建议：统一 `ReviewResult` 中间结构，两条路径都产出它。

40. **`declarative_provider.hpp` 的 `request()` 是一个 200 行的 `if-else` 方法分发**
    （`declarative_provider.cpp:164-223`），每加一个 RPC 方法就长一截。
    建议：方法表 `QHash<QString, Handler>` 或拆 `CatalogApi`/`AttemptApi`。

41. **ABI 头 `provider_abi.h` 与 host 侧 `ProviderLoader` 的函数指针类型手动
    `reinterpret_cast` 绑定**（`provider_loader.cpp:97-108`）。安全但脆弱：
    签名一旦改了 ABI 头而忘了改 loader 这边的 cast，运行时才崩。
    建议：loader 侧用与 ABI 头同源的函数指针 typedef，编译期对齐。

### D. 风险模式 / 正确性

42. **原生 provider 用 `PreventUnloadHint` 且 `unload()` 仍调 `library_.unload()`**
    `provider_loader.cpp:89` 设了 `PreventUnloadHint`（注释说为防止 Qt 注册的回调
    指向已卸载镜像），但 `unload()`（`:157`）依旧 `library_.unload()`，
    两者语义矛盾——要么真的不卸载（删掉 unload 调用），要么不要 PreventUnloadHint。
    当前实现下 `unload()` 实际可能让回调悬空。建议二选一并补测试。

43. **`request()` 里 `QTimer::singleShot(0, this, [this, response]{...})` 捕获
    `response` 值拷贝进 lambda，若 provider 在 unload 后才投递，`this` 已悬空**
    `provider_loader.cpp:169-171`。`QueuedConnection` 到 `this`，对象销毁后
    Qt 会断连，但 `declarative_.request` 是同步返回、再异步 emit；若用户在
    0ms 窗口内切题库（`loadProvider`→`unload`→`declarative_.unload()`），
    排队中的 lambda 持有旧 response 但 `this` 仍存活，会路由到错误 provider。
    建议：投递时带 generation/序号校验，丢弃陈旧响应。

44. **`QSettings` 散落全代码各处、key 字符串硬编码且无集中常量**
    如 `main_window.cpp:110,405-411,1220`，`provider_installer.cpp`、`draft_store.cpp`
    各自 `QSettings().value("…")`。key 拼写错、改名难追踪。
    建议：集中 `SettingsKeys` 常量与一个 `Settings` 封装。

45. **`QMessageBox::question` 默认按钮文案/图标在三平台不一致**
    `main_window.cpp:585,1170,1420` 等大量用 `QMessageBox::question/Yes/No`，
    macOS 上 Yes/No 会被系统改成“好/取消”风格，中文“是/否”与按钮语义偶尔错位。
    建议：显式 `setButtonText` 或改用自定义对话框统一。

46. **`answerViewportMaximumHeight` 等布局魔法数字散落**
    `main_window.cpp:1571-1575`、`:1493-1496`、`:1136-1140` 一堆 `14/7/26/28`
    等像素常量，跨平台 DPI 下易错位。建议抽布局常量并注释来源。

47. **`safeRelativePath` 用 `clean == path` 判定，对大小写/重复斜杠敏感**
    `provider_installer.cpp:31-37`。`QDir::cleanPath` 会规整 `./`，但
    `clean == path` 要求原始串已是规整形；带 `//` 或 `./` 的合法路径会被拒。
    建议：仅校验“clean 后不含 `..` 且非绝对”，不要求 `clean == path`。

48. **`document_extractor.cpp` 的 `mz_free(bytes)` 后仍构造 `QByteArray`**
    `:97-101`：`extract_to_heap` 返回 `bytes`，先 `QByteArray(bytes, size)` 拷贝，
    再 `mz_free(bytes)`。顺序正确（先拷贝后释放），但 `bytes == nullptr && size != 0`
    时会构造异常；`if (bytes || size == 0)` 的判定让“有 size 无 ptr”走 else 分支，
    返回空却不报错。建议：失败即返回 error，不要把“空”当成功。

49. **`rule_based_generator.cpp` 的 `comparableText` 用 NFKC 去标点做重复检测**
    `:126-130`：`NormalizationForm_KC` + `remove([\\s\\p{P}\\p{S}])`。
    对“下面哪项正确”与“下面哪项是正确的”这类同义不同形会漏判重复（这本就难），
    但对“1+1=2”与“1＋1＝2”这类全角差异能合并，OK。风险在 `contains(expected)`
    子串匹配（`:144`）可能误判短答案命中长选项。建议：保留但加长度阈值。

50. **测试覆盖偏向 core/sdk，UI 与生成器交互层覆盖薄**
    `tests/` 有 `bank_validator`、`rule_based_generator`、`chunker`、
    `generation_workflow`、`material_package_e2e` 等，覆盖不错；但
    `main_window` 的响应路由、`studio_window` 的向导状态机无 UI 级测试。
    建议：把 `handleProviderResponse` 等纯路由逻辑抽离后补单测。

### E. 重复 / 可复用

51. **`fail(QString* error, …)` 辅助函数在 `draft_store.cpp:14`、
    `declarative_provider.cpp:16`、`provider_installer.cpp:26` 各写一份**
    （签名略有差异）。建议提到公共 `core` 工具头。

52. **`readJson`/`readObject` 文件读取 + 大小限制模式重复**
    `provider_installer.cpp:57-63`、`declarative_provider.cpp:17-22`、
    `bank_validator` 间接。建议统一 `readJsonFile(path, maxBytes, &error)`。

53. **“逐像素/逐字符归一化”逻辑（`normalizeOptionLabel`/`normalizeAnswer`）与
    `chunker` 的选项识别正则在两处各自维护选项标记集合**
    （`rule_based_generator.cpp:65-93` vs `chunker.cpp:46-50` 的 `[A-DＡ-Ｄ]`）。
    建议抽 `option_label` 公共模块，避免一处改漏一处。
